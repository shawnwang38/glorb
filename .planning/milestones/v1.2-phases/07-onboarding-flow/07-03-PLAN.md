**Edit 4 — Add IPC handlers at the end of the file (after the existing ipcMain.handle('send-serial', ...) block):**

```javascript
// Phase 7 — ONBOARD-01/06: onboarding window lifecycle
ipcMain.handle('open-onboarding', () => {
  // Reset flag so onboarding runs from the beginning
  store.set('onboardingComplete', false)
  if (onboardingWin && !onboardingWin.isDestroyed()) {
    onboardingWin.focus()
    return
  }
  createOnboardingWindow()
})

ipcMain.handle('close-onboarding', () => {
  if (onboardingWin && !onboardingWin.isDestroyed()) {
    onboardingWin.close()
  }
})
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer → IPC → main | open-onboarding and close-onboarding channels cross the contextBridge |
| store writes from onboarding | Main process reads onboardingComplete to gate window creation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-06 | Tampering | open-onboarding IPC channel | accept | Channel only opens a BrowserWindow (no filesystem write, no privileged system call); worst case = unwanted window opens |
| T-07-07 | Denial of Service | close-onboarding called multiple times | mitigate | Handler checks `!onboardingWin.isDestroyed()` before calling close() — safe no-op on repeat calls |
| T-07-08 | Spoofing | onboardingComplete store flag | accept | Local Electron store file; no remote trust boundary; user could manually edit but that only affects their own onboarding gate |
</threat_model>

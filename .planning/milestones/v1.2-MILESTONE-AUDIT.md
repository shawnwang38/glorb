## Blocking Gaps

### 1. ONBOARD-06: Retake Test Button Unwired

**File:** `renderer.html` line 136  
**Issue:** `#btn-retake` DOM element exists but has no click event listener. `window.glorb.openOnboarding()` is exposed via preload.js and `ipcMain.handle('open-onboarding')` exists in main.js — the full IPC chain is intact inward. Only the DOM→preload connection is missing.  
**Fix:** Add one line to renderer.html's JS block:
```js
document.getElementById('btn-retake').addEventListener('click', () => window.glorb.openOnboarding())
```

### 2. INTERV-03 / INTERV-04: stopTimer Does Not Exist

**File:** `renderer.html` line ~593  
**Issue:** The `onInterventionTerminate` handler calls `stopTimer()` which is not defined anywhere. The correct function is `resetTimer()`. A `typeof stopTimer === 'function'` guard silently swallows the no-op, so the UI popup appears but the countdown timer continues running after a Weak path termination.  
**Fix:** Change `stopTimer` → `resetTimer` in the `onInterventionTerminate` callback:
```js
window.glorb.onInterventionTerminate((data) => {
  if (typeof resetTimer === 'function') resetTimer()
  showInterventionPopup(data.message)
})
```

## Advisory Findings (Non-Blocking)

### ONBOARD-03: Name Display Not Updated Post-Onboarding

`main.js` sends `onboarding-complete` event after close-onboarding IPC, but renderer.html has no `window.glorb.onOnboardingComplete(...)` listener. After onboarding completes, the timer window's name display (if any) won't refresh until app restart. Not a functional break but a UX gap.

**Fix:** Add listener in renderer.html:
```js
window.glorb.onOnboardingComplete(() => updateFocusMessage())
```
(if `updateFocusMessage()` references the userName store value)

### Phase 7: No VERIFICATION.md

Phase 7 (Onboarding Flow) was executed and shipped all 3 plan summaries, but no formal VERIFICATION.md was created. All ONBOARD requirements (except ONBOARD-06) are implemented based on SUMMARY evidence and integration checker confirmation. This is an audit artifact gap, not a code gap.

### Strength Not Persisted on First Launch

After onboarding completes, `strength` is not written to store. First-time drift routing correctly defaults to `'weak'` via `store.get('strength', 'weak')`, and the renderer UI also defaults to Weak via the HTML `active` class. The two defaults are consistent. The value is only persisted after the user explicitly clicks a strength button.

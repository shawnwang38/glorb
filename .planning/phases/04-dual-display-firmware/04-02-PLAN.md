  <what-built>
    Firmware is compiled and ready to upload (Plan 01 complete). Before upload, the hardware wiring must match the firmware pin assignments.

    Display 2 was previously wired to A4/A5 (sharing the HW I2C bus with Display 1).
    The new firmware expects Display 2 on A2 (SDA=16) and A3 (SCL=17) for software I2C.
  </what-built>
  <how-to-verify>
    On your breadboard:
    1. Locate Display 2's SDA wire — move it from A4 to A2 on the Arduino UNO
    2. Locate Display 2's SCL wire — move it from A5 to A3 on the Arduino UNO
    3. Display 1 remains on A4 (SDA) and A5 (SCL) — do not move these
    4. Confirm both displays still have 3.3V/5V power and GND connected
    5. Confirm both displays have their I2C addresses on separate buses (no address conflict possible — they're on different buses)
  </how-to-verify>
  <acceptance_criteria>
    - Display 2 SDA is connected to Arduino A2 (not A4)
    - Display 2 SCL is connected to Arduino A3 (not A5)
    - Display 1 SDA remains on Arduino A4
    - Display 1 SCL remains on Arduino A5
    - Both displays powered (3.3V or 5V + GND)
  </acceptance_criteria>
  <resume-signal>Type "wired" when Display 2 has been moved to A2/A3</resume-signal>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 3: Visual verification of both displays</name>
  <read_first>
    - .planning/phases/04-dual-display-firmware/04-CONTEXT.md (expected graphics: ellipse 40w×60h centered on 64×128 portrait canvas; "^" logisoso32 centered)
  </read_first>
  <what-built>
    Firmware is running on the Arduino. The demo loop calls:
    - showDisplay(OPEN_EYES) → both displays show filled oval → 2 second pause
    - showDisplay(SMILE) → both displays show "^" character → 2 second pause
    - Repeats indefinitely

    Expected visual on each display (after rotation correction):
    - Portrait orientation (taller than wide — effectively 64px wide × 128px tall canvas)
    - OPEN_EYES: solid filled ellipse, roughly centered, ~40px wide × 60px tall
    - SMILE: "^" caret character in large logisoso32 font, roughly centered
  </what-built>
  <how-to-verify>
    Observe both displays for at least one full cycle (4+ seconds):

    1. OPEN_EYES check (both displays simultaneously):
       - Both displays show a filled oval/ellipse shape
       - Shape is portrait-oriented and roughly centered on each display
       - No upside-down or sideways text/shapes (rotation is correct)

    2. SMILE check (both displays simultaneously):
       - Both displays show a "^" caret character
       - Character is large (logisoso32 = 32px font) and roughly centered
       - No upside-down or sideways rendering

    3. Synchronization check:
       - Both displays transition at the same time (within visual perception — ~milliseconds apart is fine)
       - Neither display lags visibly behind the other

    4. Rotation calibration (note issues for tuning):
       - If Display 1 appears upside down → note "D1 rotation wrong, try U8G2_R3"
       - If Display 2 appears upside down → note "D2 rotation wrong, try U8G2_R1"
       - If oval appears off-center → note the direction for coordinate tuning
       - Minor centering issues are acceptable — exact pixel coordinates are Claude's discretion per CONTEXT.md
  </how-to-verify>
  <acceptance_criteria>
    - Display 1 (A4/A5) shows content — not blank, not all-white
    - Display 2 (A2/A3) shows content — not blank, not all-white
    - Both displays show the same state at the same time (both oval OR both caret — never one oval and one caret simultaneously)
    - OPEN_EYES state shows an ellipse/oval shape (not text, not a box)
    - SMILE state shows a "^" character (not ">/<" pair from prior firmware)
    - States alternate every approximately 2 seconds
    - No display is rotated 180° from expected portrait orientation (if rotation is wrong, note it for a fix)
  </acceptance_criteria>
